## Интерфейсы в го

Допустим у нас есть структура BMW, полей в ней нет, только метод с ресивером по значению, который выводит только текст "я бмв 228 лошадиных сил, жмем на газ"
```go
package main

import "fmt"

type BMW struct{}

func (b BMW) StepOnGas() {
	fmt.Println("я бмв 228 лошадиных сил, жмем на газ")
}

  

func main() {
	bmw := BMW{}
	bmw.StepOnGas()
}
```
 Потом добавили еще структуру Zhiga, которая тоже не имеет полей и у которой тоже только 1 метод, который выводит "я жига , я ржавое ведро"
 ```go
 package main

  

import "fmt"

type BMW struct{}

func (b BMW) StepOnGas() {
	fmt.Println("я бмв 228 лошадиных сил, жмем на газ")
}

type Zhiga struct{}

func (z Zhiga) StepOnGas() {
	fmt.Println("я жига , я ржавое ведро")
}

  

func main() {
	bmw := BMW{}
	bmw.StepOnGas()
	
	zhiga := Zhiga{}
	zhiga.StepOnGas()
}
 ```
 теперь добавим функцию для взаимодействия с отдельно взятым автомобилем (в качестве аргумента должна  принимать экземпляр структуры)
 ```go
 func ride( /**/ ) {
	fmt.Println("Я водитель")
	fmt.Println("Я сажусь в машину")
	fmt.Println("Нажимаю на газ")
}
 ```

Вот реализация для BMW, в эту функцию еще дописываем вызов метода для экземпляра BMW
```go
func ride(bmw BMW) {
	fmt.Println("Я водитель")
	fmt.Println("Я сажусь в машину")
	fmt.Println("Нажимаю на газ")
	bmw.StepOnGas()
}

func main() {
	bmw := BMW{}
	ride(bmw)
}
```

Теперь допустим,что мы хотим сделать ту же самую функцию для жиги, получается все то же самое, просто надо скопировать  
```go
func ride(zhiga Zhiga) { // ride будет подсвечено красным
	fmt.Println("Я водитель")
	fmt.Println("Я сажусь в машину")
	fmt.Println("Нажимаю на газ")
	zhiga.StepOnGas()
}
```
Но тут сразу 2 проблемы :
1) Дублирование кода, если бы машин таких было много, то мы бы должны были это все кучу раз дублировать и менять там везде по одной строчке что-то
2)  Одинаковое название функции, такое в принципе недопустимо правилами ЯП, так как он тупо не может понять к чему конкретно ему надо будет потом обращаться
По-хорошему надо,чтобы в ride передавался не единичный автомобиль, а просто сущность, которая из себя представляет любой автомобиль, и мы бы просто передавали туда все, нежели чем меняли потом  захламляли код дубликатами

Эту проблему и решают интерфейсы. Получается,что интерфейсы в го это возможность объявить какие-то методы для той или иной сущности, но не завязываться на том как конкретно была эта сущность реализована    
Вот как это выглядит в го:
```go
type Auto interface {
	StepOnGas()
}
```

Мы объявляем интерфейс(объявление как у структуры кста) и передаем туда функции, которые будут выполняться не зависимо от привязки к тому или иному интерфейсу
Но тем не менее, при реализации через экземпляры структуры надо будет,чтобы у структуры этот метод был описан
```go
func ride(auto Auto) {
	fmt.Println("Я водитель")
	fmt.Println("Я сажусь в машину")
	fmt.Println("Нажимаю на газ")
	auto.StepOnGas()
}

```

Что конкретно произошло? Мы сказали,что хотим сделать интерфейс  и назвали его Auto,  получается,что  мы будем считать автомобилем все, где есть метод StepOnGas(), то есть, для нас главное,чтобы просто был метод StepOnGas(), что там будет происходить нам не интересно
Вот в примере ниже мы передаем экземпляр структуры bmw, хотя ride() принимает в качестве аргумента Auto, но так как мы в Auto уже описали,что считаем машиной все,что имеет метод StepOnGas(),то ошибки не возникает, так как мы можем хранить что угодно,что подходит под описание интерфейса, а в описании неоднократно сказанный выше метод
Если говорить проще, то принимается любая структура,где есть StepOnGas() :
```go
func main() {
	bmw := BMW{}
	ride(bmw)
}
```

С жигой к словцу тоже будет работать:
```go
func main() {
	zhiga := Zhiga{}
	ride(zhiga)
}
```

По итогу мы завязываемся на том,что если мы какой-то интерфейс объявляем, то можем работать с чем угодно,что подходит под этот интерфейс

## Нюансы с количеством функций в интерфейсе
Но если добавить в интерфейс еще что-то, то компилятор на это будет ругаться
```go
type Auto interface {
	StepOnGas() 
	StepOnBrake()
}

// какой-то код

func main() {
	lambo := Lambo{}
	ride(lambo) // тут уже будет ошибка
}
```

Мы буквально сказали интерфейсу,что теперь будем считать автомобилем все, у чего есть методы StepOnGas() И StepOnBrake(), тоесть, если нет хотяб одного из этих методов, то это уже не машина(проще говоря экземпляр структуры не будет удовлетворять условиям интерфейса)

Если для какой-то из наших структур объявить этот метод,то все заработает:
```go
type Auto interface {
	StepOnGas()
	StepOnBrake()
}

type BMW struct{}

func (b BMW) StepOnGas() {
	fmt.Println("я бмв 228 лошадиных сил, жмем на газ")
}
func (b BMW) StepOnBrake() {
	fmt.Println("Тормозим")
}

// остальной код

func ride(auto Auto) {
	fmt.Println("Я водитель")
	fmt.Println("Я сажусь в машину")
	fmt.Println("Нажимаю на газ")
	auto.StepOnGas()
	auto.StepOnBrake() 
}

  

func main() {
	bmw := BMW{}
	ride(bmw)
}
```

Да даже если этот метод нигде не будет использован, объявлен он должен быть

Если добавить для нашей структуры еще один метод,но не закинуть его в интерфейс,то у нас все равно будет ride() принимать наш экземпляр структуры  как аргумент, потому что главное,что остальные 2 метода описаны:
```go
type Auto interface {
	StepOnGas()
	StepOnBrake()
}

type BMW struct{}

func (b BMW) StepOnGas() {
	fmt.Println("я бмв 228 лошадиных сил, жмем на газ")
}
func (b BMW) StepOnBrake() {
	fmt.Println("Тормозим")
}

func (b BMW) BipBip() {
	fmt.Println("Бип Бип")
}

// остальной код

func ride(auto Auto) {
	fmt.Println("Я водитель")
	fmt.Println("Я сажусь в машину")
	fmt.Println("Нажимаю на газ")
	auto.StepOnGas()
	auto.StepOnBrake() 
}

  

func main() {
	bmw := BMW{}
	ride(bmw) // функция отработает
}
```
Теперь если точнее то Интерфейсы - это возможность абстрагироваться от какой-то реализации для того,чтобы писать унифицированные функции, избавляться от дублирования кода 

Все сводится к тому, что мы приходим к полиморфмизму, так как мы написали одну функцию и можем использовать ее с разными структурами  и формами 

## Интерфейс на реальном примере

Допустим мы работаем в какой-то компании и нам дали таску - сделать модуль проведения оплаты, который может проводить оплату, отдавать  информацию о произошедших операциях и отменять оплату, также этот  модуль должен уметь рабоатать с разными типами проведения оплаты(биток,карта, электронные деньги по типу PayPal)
Можно было бы это все реализовать через 3 структуры, где отдельно мы рассматриваем биток, отдельно карту, отдельно PayPal, но мы бы столкнулись с проблемой дублирования кода, да и если бы надо было еще что-то добавлять, то код просто мусорился бы 

Описывать подробно пожалуй не буду, а то итак в коде много комментов, поэтому код с пояснениями будет в конце, а пока, буду тут делать всякие ремарки

Иногда встает вопрос как в конструкторе лучше всего представить возвращаемую структуру, указателем или копией
Как я понял, если надо какие-то манипуляции делать над бд или архитектура слишком хрупкая и нам непозволительно использовать копии,короче, если объект хранит состояние, требует изменения полей, имеет большой размер или реализует интерфейс,чтобы никто не мог эти злоупотребить,то лучше использовать по указателю:
```go
type PaymentModule struct {
	paymentMethod PaymentMethod
}

  

func NewPaymentModule(paymentMethod PaymentMethod) *PaymentModule {
	return &PaymentModule{
		paymentMethod: paymentMethod,
	}
}
```

Еще по ходу разработки встал вопрос, мол что лучше подойдет для хранения информации и работы с ней, коротко говоря сильных отличий нет, НО так как в методе Get для слайса используется цикл, а в мапе это все делается возвратом значения, то оптимальнее для такого юзать мапу:
```go
type PaymentInfo struct {
	ID int
	Description string
	Usd int
	Cancelled bool

}

type PaymentModuleWithSlice struct {
	s []PaymentInfo
}

  

func (m *PaymentModuleWithSlice) AddInfo(info PaymentInfo) {
	m.s = append(m.s, info)
}

  

func (m *PaymentModuleWithSlice) GetInfo(id int) PaymentInfo {
	for _, info := range m.s {
		if info.ID == id {
			return info
		}
	}
	return PaymentInfo{}
}

  

// --------

type PaymentModuleWithMap struct {
	m map[int]PaymentInfo
}

  

func (m *PaymentModuleWithMap) AddInfo(info PaymentInfo) {
	m.m[info.ID] = info
}

  

func (m *PaymentModuleWithMap) GetInfo(id int) PaymentInfo {
	info, ok := m.m[id]
	if !ok {
		return PaymentInfo{}
	}
return info
}

func main() {
	pSlice := PaymentModuleWithSlice{}
	pMap := PaymentModuleWithMap{
	m: make(map[int]PaymentInfo),
	}
//--------------------------
	iterations := 1000
	before := time.Now()
	for i := 0; i < iterations; i++ {
		info := PaymentInfo{
			ID: i,
			Description: strconv.Itoa(i),
		}
		pSlice.AddInfo(info)
	}
	fmt.Println("slice", time.Since(before))

	before = time.Now()

	for i := 0; i < iterations; i++ {
		info := PaymentInfo{
			ID: i,
			Description: strconv.Itoa(i),
		}
		pMap.AddInfo(info)
	}
	fmt.Println("map: ", time.Since(before))
//--------------------------

	before = time.Now()
	for i := 0; i < iterations; i++ {
		info := pSlice.GetInfo(i)
		_ = info
	}
	fmt.Println("slice", time.Since(before))
	before = time.Now()
	for i := 0; i < iterations; i++ {
		info := pMap.GetInfo(i)
		_ = info
	}
	fmt.Println("map: ", time.Since(before))
```

![[map_and_slice_time.png]]

Да и на скрине выше видно,что мапа в аспекте поиска быстрее, все потому что операция поиска у слайса делает все это за линейное время O(n),где n - число элементов, и тут уже идет прямая зависимость от числа элементов слайса; у мапы же дела в этом плане лучше, ведь она все делает за константное время O(1), то есть, за заданное и получается,сколько бы элементов не было, время не поменяется
![[time_complication_of_structures.png]]

Слайс очень хорошо подходит, если нужно сохранить четкий порядок элементов, например при сортировке там чего-то, в мапе это сделать нереально 

Когда реализуем метод,который возвращает информацию о всех элементах мапы очень хочется сделать так :
```go
func (p PaymentModule) AllInfo() map[int]PaymentInfo {
	return p.paymentsInfo
} 
```
Но так делать нельзя, так как  мапа имеет  указатель на какие-то данные, то при таком методе возвращается его копия, которая также указывает на эти данные и потенциально через эту копию можно менять что-то в мапе, а оно нам не надо, поэтому надо возвращать не просто копию мапы, а копию мапы которая указывает на копию данных(многоходовочка):
```go
func (p PaymentModule) AllInfo() map[int]PaymentInfo {
	tempMap := make(map[int]PaymentInfo, len(p.paymentsInfo))
	for k, v := range p.paymentsInfo {
		tempMap[k] = v
	}
	return tempMap
} // о всех операциях
```
Мы тут создали новую мапу,через цикл записали в нее ключи и значения старой и потом вернули ее 

Теперь к описанию проги(сразу оговорюсь, описывать буду по пакетам, как только описание одного кончается, начинается описание другого):
Для начала думаем, что может повторяться, в нашем случае методы оплаты, говорим то, что метод оплаты считается таковым,если у него есть методы Pay(usd int)int и Сancel(id int)
```go
package payments
type PaymentMethod interface {
	Pay(usd int) int
	Cancel(id int)
}
```
Далее создаем общую структуру модуля, в качестве полей передаем ВНИМАНИЕ экземпляр структуры, который удовлетворяет условиям интерфейса PaymentMethod и мапу для хранения инфы об операциях(почему именно мапа описано выше)
```go
type PaymentModule struct {
	paymentsInfo map[int]PaymentInfo
	paymentMethod PaymentMethod
}

  

func NewPaymentModule(paymentMethod PaymentMethod) *PaymentModule {
	return &PaymentModule{
	paymentsInfo: make(map[int]PaymentInfo),
	paymentMethod: paymentMethod,
	}
}
```

Кстати, напомню, чтобы мапа работала ее надо инициализировать,что в конструкторе и описано
Далее реализованы методы,что они делают в комментах написано, от себя добавлю,что методы Cancel()  и Pay() которые описаны ниже и которые методы PaymentModule не те же самые методы, которые описаны в их телах:
```go
// Метод Pay()
// Принимает:
// 1. Описание проводимой оплаты
// 2. Сумма оплаты
// Возвращает:
// ID проведенной операции
func (p PaymentModule) Pay(description string, usd int) int {
	// 1. проводить оплату
	// 2. получать id проведенной оплаты
	id := p.paymentMethod.Pay(usd)
	info := PaymentInfo{
		Description: description,
		Usd: usd,
		Cancalled: false,
	}
	// 3. сохранить инфо о проведенной операции
	// -- описание операции
	// --сколько было потрачено
	// -- отмененная лм операция
	p.paymentsInfo[id] = info
	// 4. возвращать id проведенной оплаты
	return id
}

// Метод Cansel()
// Принимает:
// 1. ID операции
// Возвращает:
// --ничего
func (p PaymentModule) Cancel(id int) {
	info, ok := p.paymentsInfo[id]
	if !ok {
		return
	}
	p.paymentMethod.Cancel(id)
	info.Cancalled = true
	p.paymentsInfo[id] = info
}

  

// Метод Info()
// Принимает:
// 1. ID операции
// Возвращает:
// информацию о проведенной операции

func (p PaymentModule) Info(id int) PaymentInfo {
	info, ok := p.paymentsInfo[id]
	if !ok {
		return PaymentInfo{}
	}
	return info
} // о конкретной операции

// Метод AllInfo()
// Принимает:
// -- ничего
// Возвращает:
// информацию о всех проведенных операциях
func (p PaymentModule) AllInfo() map[int]PaymentInfo {
	tempMap := make(map[int]PaymentInfo, len(p.paymentsInfo))
	for k, v := range p.paymentsInfo {
		tempMap[k] = v
	}
	return tempMap
} // о всех операциях
```
