Паники отличаются от ошибок тем, что представляют из себ такие нарушения в видах работы, которые никак нельзя исправить
Допустим у нас есть код ниже:
```go
a := 0
b := 1 / a
fmt.Println(b)
```

как ожидаемо, выскочит ошибка:
```bash
panic: runtime error: integer divide by zero
goroutine 1 [running]:
main.main()
        /home/vagab/Документы/go_study/13_panic/main.go:55 +0x9
exit status 2
```
Нам пишет,что произошло деление на 0, а этого допускать нельзя
То есть, если ошибки там можно как-то обработать, то паники говорят о том,что код в принципе написан неверно, и это чревато тем,что при ошибке там условности нарушены будут, при панике программа ляжет

Ну или если что-то такое писать :
```go
slice := []int{1, 2, 3}
fmt.Println(slice[4])
```

То выведет:
```bash
panic: runtime error: index out of range [4] with length 3
goroutine 1 [running]:
main.main()
        /home/vagab/Документы/go_study/13_panic/main.go:55 +0x9
exit status 2
```
А ошибка говорит о том, что у нас нет индекса = 4

Но язык Go позволяет нам обрабатывать паники таким образом:
```go
defer func() {
	p := recover()
	if p != nil {
		fmt.Println("Была паника", p) // вывод: Была паника runtime error: index out of range [4] with length 3
	}
}()
```

Как я понял у нас после отлова паники, она кладется в recover(), оттуда в p, потом  p сравнивается с nil и если nil не равно p,то тело ветвления срабатывает

Так случается,потому что при обнаружении паники приложение схлопывается, единственное,что выполняется во время этого падения - функции в стеке отложенных вызовов(которые кладутся туда с помощью defer). Если написать recover следующей строчкой после проблемного кода, то программа до нее никогда не дойдет, поэтому она и в defer

recover() возвращает пустой интерфейс и туда кладется что угодно, если паника вызвана самой системой, то туда кладется системный объект с описанием проблемы
