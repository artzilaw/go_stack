А что же такое пакет?
Грубо говоря, пакет это папки внутри модуля
![[package.png|601]]
Получается, что пакет, это папка,в которой лежат файлы на одном уровне(короче, папка в файлами go)

На уровне пакета файлы друг другу доверяют и все друг у друга видят, но если мы хотим что-то из файла с одного пакета передать в файл с другим пакетом, то в первом пакете уже сможем разделять,что могут видеть файлы из других пакетов

## Как работать с модулями?
Для начала в директории,которую надо объявить модулем, создаем go.mod файл через команду "go mod init название_модуля)

Далее для примера можно создать папку greeting и в ней файл greeting.go(название может быть любое), в итоге получается как-то так
![[modules_practick.png]]
greeting.go:
```go
package greeting

import "fmt"

func SayHello() {
	fmt.Println("Hello golang")
}
```

main.go:
```go
package main

import "modules_and_packages/greeting"

func main() {
	greeting.SayHello()
}
```
Как мы видим, у нас импортировался пакет, как только мы этот пакет вызвали в файле main.go

**==Примечание:==** у меня почему-то не работал код из greeting, немного поискал и как оказалось в гошке называть с маленькой буквы(было sayHello) можно только приватные функции(которые работает только внутри своего пакета), поэтому ее надо называть с большой

## Несколько файлов в пакете

Давайте в папке greeting создадим еще один файл bad.go(название может быть любое), так вот,чтобы он нормально инициализировался, надо чтобы там был объявлен тот же пакет,что и в первом файле, ведь в одной папке(которую go читает как пакет) не может быть объявлено 2 пакета
![[modules_practick_bad.png.png]]
И того код получается такой:
```go
bad.go:

package greeting

import "fmt"

func SayBad() {
	fmt.Println("плохое")
}

main.go(только изменения):
greeting.SayBad()
```

То есть,чтобы из файла bad.go взять функцию,нам надо не к файлу обращаться или чему-то еще, а к **пакету** (в данном случае greeting)

## Область видимости пакета 
Файлы пакета могут видеть все  друг у друга друга, тоесть, что написано в одном файле пакета, доступно и в других файлах пакета

Создадим еще один файл под названием int.go и напишем в нем следующее:
```go
package greeting

func GiveMeInt() int {
	return 55
}
```
 Чтобы в main() эту функцию использовать,надо писать что-то вроде:
 ```go
i := greeting.GiveMeInt()
fmt.Println(i) // выведет 55
 ```

Но если в условном greeting.go не надо это все объявлять, для примера можно написать что-то вроде:
```go
package greeting

import "fmt"

func SayHello() {
	fmt.Println("Hello golang")
	j := GiveMeInt() // написали без greeting.
	fmt.Println("from SayHello:", j)
}
```

И при вызове с main()  у нас помимо текста "Hello golang",  выведется еще и j=55

## Инкапсуляция через модули

Создадим теперь пакет user с файлом user.go , в этом файле опишем структуру User по следующим полям:
```go
package user

type User struct {
	Name string
	Age int
}
```

В лекции про [[07_Структуры|структуры]] обговаривался только один метод инкапсулции(через конструктор), но это можно делать и через примечание, которое я оставлял выше

То есть, если мы не хотим,чтобы кто-то условно поменял значение какого-то поля у переменной с заданной структурой,то просто пишем с маленькой буквы имя поля и все, оно доступно только в рамках пакета user
```go
package user

type User struct {
	Name string
	age int
}
```
Таким образом, человек может как- то изменить поле Name,  но не поле age

Получается, что поля структуры с маленькой буквы,тоесть к ним не обратиться вне пакета user, остается создавать user'a только через конструктор :
```go
user/user.go
func NewUser(name string, age int) User {
	if name == "" {
		return User{}
	}
	if age <= 0 || age >= 150 {
		return User{}
	}
	return User{
		name: name,
		age: age,
	}
}

```

И получается,что ему остается только через методы менять значения полей переменной,и то в методах мы уже можем это все валидировать, примеры функций:
```go
func (u *User) SetNewName(name string) {
	if name != "" {
		u.name = name
	}
}

  

func (u *User) SetNewAge(age int) {
	if age > 0 && age < 150 {
		u.age = age
	}
}
```

Если кому-то надо посмотреть значение того или иного поля, можно создать геттеры:
```go
func (u *User) GetName() string {
	return u.name
}

func (u *User) GetAge() int {
	return u.age
}
```

И получается так,как в гетерах нет никаких указателей, потенциальный другой человек, который будет работать с кодом получит только копии, изменить он сможет только через описанные нами методы, и то там прописана валидация

И того в main.go работа со всем этим будет выглядеть так:
```go
package main

import (
"fmt"
"modules_and_packages/user"
)

func main() {
	u := user.NewUser("Мага", 20)
	u.SetNewAge(20)
	u.SetNewName("Арцик")
	fmt.Println(u.GetAge())
	fmt.Println(u.GetName())
}
```

Опять же, менять можно только через методы, но никак напрямую, чтобы не было возможности сломать

Но где-то так не принято писать, где-то принято, все зависит от команды короче, но хорошей практикой является сокрытие полей и доступ у полям только через методы,создание той или иной сущности через конструктор(ловлю референсы с C/C++, правда забыл уже все там)

Также хорошо называть функции с маленькой буквы, если нам эти функции нужны будут только в рамках одного пакета, а не других:
```go
func cho-to(){
	//что-то делает в рамках его пакета, в других он просто бесполезен
}
```

## Зачем вообще все это надо и про концепцию модуля

Да все просто, чтобы не было каши , в теории можно все это описывать в main(), но там будет кода на десятки тысяч строк, в теории можно все в один пакет закинуть,но все в итоге придет к тому,что будет выглядеть так:
![[files_without_packetges.png]]

чтобы этого избежать, в разных папка описаны файлы,отвечающие за конкретный функционал, чтобы не захламлять другие файлы, чтобы поделить программу на какие-то логические части 

Что же насчет модуля, то так как наша программа представляет из себя модуль, мы можем использовать другие модули.

Возьмем пример с библиотекой для красивого вывода структур: https://pkg.go.dev/github.com/k0kubun/pp/v3#section-readme, чтобы ее импортировать, мы должны зайти в директорию с go.mod файлом и написать в консоли:
```bash
go get github.com/k0kubun/pp/v3
```
Вместо "github.com/k0kubun/pp/v3" может быть любая другая библиотека

После этого файл go.mod выглядит так:
```go
module modules_and_packages

go 1.26.5

require (
	github.com/k0kubun/pp/v3 v3.5.2 // indirect
	github.com/mattn/go-colorable v0.1.15 // indirect
	github.com/mattn/go-isatty v0.0.20 // indirect
	golang.org/x/sys v0.29.0 // indirect
	golang.org/x/text v0.38.0 // indirect
)
```

То есть, у нас скачались файлы в язык go, где-то закешировались и через go.mod добавились зависимости  

 Если надо поменять версию,то меняем(первая строчка после require ()
 
 Также мы можем убрать зависимости из go.mod, если мы библиотеку не используем, это делается через команду:
 ```bash
 go mod tidy
 ```
 Она проверяет используем ли мы эту библиотеку и если нет, то она подчищает все зависмости, но если ее опять написать, то библиотека вернется в go.mod файл
 ```go
 package main
import (
	"modules_and_packages/user"
	"github.com/k0kubun/pp/v3"
)
func main() {
	u := user.NewUser("Мага", 20)
	pp.Println(u)
}
 ```
